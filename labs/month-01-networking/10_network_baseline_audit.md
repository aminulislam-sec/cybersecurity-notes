# Lab 10 – Network Baseline Audit: Gateway, DNS, and Public IP

## What This Lab Is About

In every lab before this one, I was looking at the network through a
specific lens — Nmap for finding machines, ss for listing services,
Wireshark for reading packets. This time I stepped back and asked something
more basic: do I actually know what each of my machines knows about its
own network? Where does it send its traffic? Who does it ask for DNS? And
what does the outside internet see when it makes a connection?

I used three commands to find out. One of them led me to an unexpected
problem I had not anticipated, which turned into the most useful part of
the whole session.

---

## A Simple Way to Think About This

Before I do any work in a new place, I like to walk around and get my
bearings — find the entrance, figure out the layout, understand how things
connect. I do not install anything or fix anything. I just learn where
everything is.

That is what this lab is. A walkthrough of my own network, to make sure
I actually understand the place I have been working in for the past nine
labs.

## Commands I Used

```bash
# Shows everything about my network connection in one place
nmcli device show

# Asks the internet what IP address it sees me coming from
curl ifconfig.me

# Shows how my machine has DNS configured at the file level
cat /etc/resolv.conf
```

## What Went Wrong First — and What It Taught Me

When I ran `nmcli device show` on Kali for the first time in this session,
something looked off immediately. The IP addresses on my two machines did
not match the way they had in every previous lab.

```
Kali:   192.168.161.128     gateway: 192.168.161.2
Mint:   10.244.116.76       gateway: 10.244.116.203
```

They were on completely different subnets. I sat with this for a moment
because it made no sense to me at first. These two machines had been on
the same network in Labs 01 through 09. Nothing about the physical setup
had changed. Same laptops, same hotspot, same room.

Then I realised what had happened. VMware had switched to a different
network mode without me noticing.

VMware gives you two main ways to connect a virtual machine to the outside
world. I think the easiest way to picture the difference is this.

Imagine I am renting a room in a shared house. In **bridged mode**, I get
my own letterbox at the front door — the postman knows I exist, my
housemates can knock on my door directly, and I am a real presence in the
building. In **NAT mode**, all my mail goes through the landlord. The
postman only knows the landlord's address. My housemates cannot reach me
directly unless the landlord passes the message along.

Bridged mode puts Kali directly on the hotspot network, just like Mint.
The hotspot gives Kali its own IP address and everyone on the network can
see it. NAT mode hides Kali behind Windows. The hotspot only sees Windows.
Kali gets its own separate private IP range that VMware creates in software
— in this case 192.168.161.x — and connects to the internet through
Windows as an intermediary.

When I looked at VMware's settings, I found it had switched to NAT at
some point between sessions. I do not know exactly when. That is the
uncomfortable part — I had not checked, so I had not noticed.

I think if I had just seen the different subnets and moved on without
investigating, every lab after this involving two machines talking to each
other would have quietly broken. Nmap would not find Mint from Kali. Ping
would fail. Netcat tests would give wrong results. And I might have blamed
the tools or my commands rather than a one-click setting in VMware that
was hiding in the background. So I am glad I caught it.

The fix was simple: VM > Settings > Network Adapter > switch from NAT back
to Bridged. Restarted Kali, ran `nmcli` again, and both machines were back
on the same subnet.

## Reading What the Output Actually Means

### The completed baseline table

Once both machines were in bridged mode, I filled in the baseline table
I had been building toward across all ten labs.

```
                    Kali Linux              Linux Mint
Private IP:         10.244.116.36           10.244.116.76
Subnet:             /24                     /24
Gateway:            10.244.116.203          10.244.116.203
DNS server:         10.244.116.203          10.244.116.203 (see below)
Public IP:          123.253.135.143         123.253.135.143
Interface:          eth0                    wlp3s0
Interface type:     Ethernet (VMware)       Wi-Fi (physical)
```

Everything lines up the way it should. Same subnet, same gateway, same
public IP.

### Why both machines show the same public IP

This was one thing I found genuinely interesting. Kali's private IP is
10.244.116.36 and Mint's is 10.244.116.76. They are different. But when
I asked the internet what address it saw me coming from, both machines
gave back the exact same answer: 123.253.135.143.

The reason is that private IP addresses like 10.x.x.x are only visible
inside the local network. When my traffic leaves the hotspot and goes out
to the internet, the hotspot replaces my private IP with its own single
public address. Everything connected to that hotspot — Kali, Mint, and
anything else — appears to the outside world as that one address. This is
called NAT, Network Address Translation, and it is how almost every home
and mobile network in the world works.

I think of it like a block of flats with one shared letterbox on the
street. From outside the building, everyone's mail comes from the same
address. Inside, each flat has its own number. The difference only matters
to people already inside the building.

### Why DNS looks different on the two machines

Both machines ultimately send their DNS queries to the same place — the
hotspot at 10.244.116.203. But they get there differently, and I could
see that from the resolv.conf files.

On **Kali**, the file said `nameserver 10.244.116.203` directly. When my
browser or any other program asks "what is the IP address for google.com",
that question goes straight to the hotspot. Simple and direct.

On **Mint**, the file said `nameserver 127.0.0.53`. That address is not
a real server anywhere on the network. It is a local service running on
Mint itself, called systemd-resolved. I actually spotted this in Lab 08
when it appeared as a listening service on port 53. At the time I noted
it was localhost-only and moved on. Now I understand what it does.

Mint sends DNS queries to its own local resolver first. That resolver
checks whether it already has a cached answer from a recent query. If it
does, it answers immediately without bothering the hotspot. If it does
not, it forwards the question to 10.244.116.203 and gets the answer from
there. The result is the same either way, but Mint's approach is one step
more sophisticated because it can answer repeated queries much faster from
its own memory.

```
Kali DNS path:
my program → hotspot (10.244.116.203) → answer

Mint DNS path:
my program → local stub (127.0.0.53) → hotspot (10.244.116.203) → answer
                         ↑
               checks its cache first
```

### What the interface types tell me

Kali shows `GENERAL.TYPE: ethernet` even though there is no physical
ethernet cable anywhere near it. VMware presents the virtual network
connection to Kali as if it were a wired ethernet card. Kali cannot tell
the difference. From its perspective, it is plugged in with a cable.

Mint shows `GENERAL.TYPE: wifi` because it is a physical machine with a
real wireless card connecting to the hotspot through the air.

This is part of why Kali's latency to the gateway was slightly lower than
Mint's in Lab 07. A virtual cable that never leaves the same physical
machine is faster than a real wireless signal crossing the room.

### What the connection name on Mint means

The `nmcli` output for Mint showed `GENERAL.CONNECTION: Auto Shohail1`.
That is the name of the saved Wi-Fi profile that Mint created when this
hotspot was first connected to. The "Auto" means Mint joins it
automatically whenever it is in range. Nothing significant here — just
a personal label that Mint stored from the first connection.

## A Record of the NAT Session

I want to keep the NAT mode results in this note rather than pretending
that session did not happen. Here is what Kali looked like before I
switched back to bridged mode.

```
Kali in NAT mode:
Private IP:     192.168.161.128/24
Gateway:        192.168.161.2        (VMware virtual — does not really exist)
DNS:            192.168.161.2
Public IP:      123.253.135.143      (same, because it still exits through Windows)
```

The gateway 192.168.161.2 is not a real device. VMware creates it in
software to give Kali something to route through. I could ping it and it
would respond, but it lives only inside VMware's virtual network. The
public IP was still the same because Kali was still reaching the internet
— just through Windows as an invisible middleman rather than directly
through the hotspot.

In NAT mode, Mint could not see Kali at all. The hotspot could not see
Kali. For any lab where the two machines need to find each other, NAT mode
quietly breaks everything. Bridged mode is the right setting for this work.

## How This Connects to Previous Labs

Lab 01 is where I first read IP addresses manually from each machine. This
lab did the same thing more efficiently with `nmcli`, and added the public
IP layer that Lab 01 never covered.

The DNS stub on Mint at 127.0.0.53 first appeared in Lab 08 as a listening
service I noted but did not fully explain. This lab closed that gap — I
now know what it is doing and where it fits in the chain.

The public IP 123.253.135.143 was invisible in Lab 09 but was present in
every packet Wireshark captured. When Kali visited Google, Google saw that
address — not 10.244.116.36. This lab confirmed that number and explained
why.

And the NAT finding is a reminder I want to carry forward into every future
lab: the environment matters as much as the commands. A tool can work
perfectly and still give me wrong results if the layer underneath it is
not what I think it is. Checking the baseline at the start of a session
is not just a good habit. It is the first step of any honest experiment.
